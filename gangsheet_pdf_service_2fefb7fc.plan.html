Plan: Gangsheet PDF Service

Xây dựng BE service gồm API upload ảnh (lưu nội bộ), API thay thế bản đã chỉnh sửa, và API render PDF gang-sheet từ layout do FE nesting cung cấp. BE không làm bin-packing — trách nhiệm tiết kiệm film thuộc FE.

Greenfield · DTF gang-sheet · PNG / WEBP / AVIF · ảnh lớn 20–100 MB

Checklist triển khai

Bối cảnh nghiệp vụ

Dịch vụ in DTF (Direct-to-Film) cần xuất một tấm film (gang-sheet) chứa nhiều thiết kế đã được xếp chồng để in một lần. Chiều rộng film cố định theo máy in (mặc định 22 inch); chiều dài phụ thuộc cách xếp ảnh. File thiết kế thường rất lớn (20–100 MB) do độ phân giải cao phục vụ in.

3. Nesting trên FE

FE nhận danh sách imageId + kích thước in mong muốn (inch) từ user. Công cụ nesting:

  • Xếp các item trên canvas rộng 22 inch (printable width có thể trừ padding)
  • Tính x, y (góc trên-trái), width, height (inch trên film), rotation (độ)
  • Expand quantity thành nhiều item riêng hoặc gửi kèm quantity — chốt 1 quy ước với FE
  • Tính sheet.heightInch = chiều dài film sau nesting (đáy xa nhất + margin)

5. Tải PDF

GET /api/gang-sheets/{id}/download — stream file PDF đã tạo. PDF có TTL (configurable); job nền dọn file hết hạn.

API contract

1. Upload ảnh

POST /api/images — multipart/form-data, field file

  • Validate: format (PNG/WEBP/AVIF), max size/file (hỗ trợ 20–100 MB)
  • Đọc metadata (width, height) lúc upload — fail sớm nếu corrupt

Response 201:

{
  "id": "uuid",
  "filename": "design-01.png",
  "width": 4800,
  "height": 3600,
  "format": "PNG",
  "sizeBytes": 52428800,
  "version": 1
}

POST /api/images/{id}/replace — team upload bản đã sửa

  • Giữ cùng id để FE nesting không phải remap
  • Tăng version, invalidate cache decode cũ

GET /api/images/{id} — metadata
GET /api/images/{id}/download — stream file

Luồng kỹ thuật render PDF

flowchart TD A[POST /api/gang-sheets/pdf] --> B[Validate request schema] B --> C[Validate từng item] C --> D[Resolve ImageAsset theo imageId] D --> E[Validate layout trên sheet] E --> F[Parallel load ảnh unique] F --> G[Budget tổng size bytes] G --> H[PDDocument render] H --> I{Với mỗi item} I --> J[Tính decodeSize từ inch × DPI] J --> K[Decode subsampled / ffmpeg AVIF] K --> L[Embed JPEG hoặc lossless nếu alpha] L --> M[Draw với matrix rotation] M --> N[Save PDF atomic write-then-move] N --> O[Trả response + warnings]

Quy tắc validate layout (BE)

  • sheet.widthInch khớp config máy in (vd 22")
  • sheet.heightInch do FE gửi sau nesting — BE validate items nằm trong sheet
  • Mỗi item: width, height > 0; rotation hợp lệ
  • x >= 0, y >= 0; bounding box sau rotation nằm trong sheet
  • imageId tồn tại, status ACTIVE
  • Tổng item ≤ limit; tổng dung lượng ảnh unique ≤ budget
  • Không validate overlap — FE chịu trách nhiệm nesting

Quy tắc decode / chất lượng in

  • renderDpi (vd 300): decodeWidth = min(imageWidthPx, ceil(item.widthInch × renderDpi))
  • Cap maxRasterPixels tránh OOM với ảnh 20–100 MB
  • Subsampling khi decode PNG/WEBP
  • AVIF: ffmpeg hoặc ImageIO plugin + libavif — document dependency deploy
  • Warning nếu effective DPI < ngưỡng (vd 150) — không block render

PDF kỹ thuật

  • Apache PDFBox 3.x
  • 1 page = 1 gang-sheet; page size = sheet.widthInch × sheet.heightInch (points = inch × 72)
  • Gốc tọa độ: top-left, đơn vị inch → convert PDF points, flip Y
  • Rotation quanh tâm item
  • Alpha: LosslessFactory; không alpha: JPEGFactory ~0.95
  • Cache PDImageXObject theo (imageId, version, decodeSize)

Bảo mật & vận hành

  • Auth: API key / JWT giữa FE và BE
  • Upload: whitelist MIME + magic-byte detect
  • Path traversal: chỉ imageId UUID, không nhận raw path từ client
  • Rate limit upload + render
  • Idempotency key trên POST /api/gang-sheets/pdf (optional)
  • Logging & metrics: latency, decode failures, PDF size, OOM guard
  • JVM heap sizing cho workload ảnh lớn; giới hạn concurrency decode/ffmpeg

Triển khai theo phase

Phase 1 — MVP (2–3 sprint)

  • Upload API + local storage + metadata DB
  • Replace API (cùng imageId)
  • Render PDF API nhận layout từ FE
  • PNG + WEBP + AVIF
  • Download PDF + validation + DPI warnings
  • Test E2E với fixture 20–100 MB

Phase 2 — Production hardening

  • Docker/deploy docs (storage, ffmpeg, JVM heap)
  • Auth, rate limit, TTL cleanup
  • Concurrency limit decode
  • Integration tests fixture 50–100 MB

Phase 3 — Scale

  • Object storage + presigned upload
  • Async render queue nếu PDF > 30s
  • Multi-tenant, audit log version ảnh

Công nghệ đề xuất

  • Runtime: Java 21 LTS + Spring Boot 3
  • PDF: Apache PDFBox 3.x
  • WEBP: webp-imageio
  • AVIF: ffmpeg on server hoặc ImageIO plugin + libavif
  • DB: PostgreSQL; H2 cho dev
  • Storage: Local FS → S3
