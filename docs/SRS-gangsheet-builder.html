1. Giới thiệu

1.1 Mục đích

Tài liệu mô tả yêu cầu phần mềm cho chức năng xây dựng gang sheet DTF theo mô hình Ninja Transfers: user thiết kế layout trên canvas 22", lưu design, thêm vào giỏ, checkout; backend lưu snapshot và chốt design khi đặt hàng để in.

1.2 Phạm vi

Trong phạm vi (MVP)
  • Builder config, Save draft, Save & Add to Cart
  • Load / list design
  • Cart CRUD
  • Checkout → order
  • Ops render PDF (nội bộ)
Ngoài phạm vi (Phase 2+)
  • Upload ảnh qua BE · Gallery ảnh · Auto-nest trên BE
  • Preview PDF cho user · Duplicate / delete design
  • User auth / multi-tenant · Pricing · Payment gateway
  • Stock art, text tool BE
Phân công: FE xử lý upload S3, gallery, nest/drag canvas. BE nhận snapshot cuối (sheet + items).

2. Mô tả tổng quan

2.1 Bối cảnh hệ thống

flowchart LR FE[Builder FE] <-->|HTTPS| API[Spring Boot API] API <-->|JDBC| DB[(H2 / PostgreSQL)] FE -->|upload| S3[(S3 / CDN)] API -->|fetch URL| S3 OPS[Ops / Production] --> API

2.2 Lớp người dùng

Ops / ProductionNhận order, render PDF, in DTF, cập nhật trạng thái đơn
FE DeveloperTích hợp builder với API

2.3 Ràng buộc

  • Chiều rộng sheet cố định 22 inch (app.pdf.sheet-width-inch)
  • Ảnh truy cập qua URL công khai (S3/CDN) trong items[].img
  • BE không validate layout / nest khi save (tin snapshot FE)
  • BE ép sheet.width_inch từ config khi persist
  • Database dev: H2 file; production: PostgreSQL (tương lai)

2.4 Giả định

  • FE upload ảnh lên S3 trước khi save; snapshot chỉ chứa URL items[].img
  • FE nest/drag trên canvas, gửi sheet.height_inch sau khi layout xong
  • Guest-first: user không bắt buộc sign in/sign up; FE giữ design_id trong localStorage
  • Cart global per BE instance (MVP) — Phase 2: X-Guest-Session-Id hoặc user auth
  • User có thể sửa draft kể cả sau Save & Add to Cart; chỉ khóa khi checkout
  • Success: ApiResponse (HTTP 2xx). Failure: ErrorResponse (HTTP 4xx/5xx). Cả hai có trace_id; phân biệt nghiệp vụ qua code (MGS_xx)

3. Luồng nghiệp vụ — DTF Gang Sheet Builder

6. Đặc tả API

Tài liệu chi tiết contract REST cho MVP. HTTP 2xx → ApiResponse<T>; HTTP 4xx/5xx → ErrorResponse. JSON field dùng snake_case trừ khi ghi chú alias.

6.1 Envelope — Success vs Error

Hai type tách riêng. Client có thể gửi header X-Trace-Id; nếu thiếu, server tự sinh và echo lại trong header + body.

ApiResponse — HTTP 200/201 (một số POST trả 201, status trong body khớp HTTP):

{
  "status": 200,
  "code": "MGS_30",
  "message": "Gang sheet saved as draft",
  "trace_id": "8d2f7a98b3c94d22",
  "timestamp": "2026-06-29T10:00:00.000000Z",
  "data": { }
}

ErrorResponse — lỗi đơn (success=false, không có data). error là tên HTTP status (vd. BAD_REQUEST); phân loại nghiệp vụ qua code (MGS_xx):

{
  "success": false,
  "status": 404,
  "error": "NOT_FOUND",
  "code": "MGS_34",
  "message": "Gang sheet not found: uuid",
  "trace_id": "8d2f7a98b3c94d22",
  "timestamp": "2026-06-29T10:00:00.000000Z"
}

Lỗi validation — HTTP 400, code=MGS_3, kèm details[]:

{
  "success": false,
  "status": 400,
  "error": "BAD_REQUEST",
  "code": "MGS_3",
  "message": "Validation failed",
  "trace_id": "8d2f7a98b3c94d22",
  "timestamp": "2026-06-29T10:00:00.000000Z",
  "details": [
    { "code": "MGS_3", "field": "items", "message": "items must not be empty" }
  ]
}

10. Tiêu chí chấp nhận

AC-01 — Save draft mới

  • GIVEN canvas có ≥1 item
  • WHEN POST /api/gang-sheets không có design_id
  • THEN 200, status=draft, design_id mới, snapshot khớp request

AC-02 — Upsert draft

  • GIVEN design draft tồn tại
  • WHEN POST với cùng design_id và items mới
  • THEN snapshot cập nhật, updated_at đổi

AC-03 — Save & Add to Cart (lần đầu)

  • WHEN POST /api/gang-sheets/cart không gửi quantity
  • THEN design in cart, cart_quantity=1

AC-03b — Đổi quantity trong cart

  • WHEN PATCH /api/cart/items/{line_id} với quantity=3
  • THEN cart_quantity=3; design snapshot không đổi

AC-04 — Edit sau add cart

  • GIVEN design trong cart qty=3, status draft
  • WHEN POST /api/gang-sheets/cart lại (không gửi quantity)
  • THEN snapshot cập nhật; cart_quantity vẫn = 3

AC-05 — Checkout

  • GIVEN cart có 2 lines
  • WHEN POST /api/orders
  • THEN order submitted, 2 order lines có snapshot, designs confirmed, cart empty

AC-06 — Không sửa confirmed

  • GIVEN design confirmed
  • WHEN POST /api/gang-sheets với design_id
  • THEN 409 MGS_35

AC-07 — Reload design

  • WHEN GET /api/gang-sheets/{id}
  • THEN trả đủ sheet, items, in_cart, cart_quantity

AC-08 — Remove khỏi cart xóa design

  • GIVEN cart có builder line gắn design_id draft
  • WHEN DELETE /api/cart/items/{line_id}
  • THEN line biến mất; gang_sheets.is_deleted=true, deleted_at được set; GET /api/gang-sheets/{design_id} → MGS_34; POST /api/gang-sheets với cùng design_id → MGS_34; user phải tạo design mới (không gửi design_id) để add cart lại
